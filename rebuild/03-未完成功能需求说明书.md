## 3.1 WebDAV 集成（UI 存在但无后端）

### 现状
`ProfilePage.jsx` 中有 WebDAV 配置 UI（URL、用户名、密码输入框），但：
- 密码加密/解密逻辑（旧版 AES-CBC `crypto.ts`）已随 Next.js 版删除
- 无 WebDAV 上传 API（旧版 `/api/upload-to-webdav` 已删除）
- `user_settings` 表中有 `webdav_url`, `webdav_user`, `webdav_pass_enc` 字段但迁移脚本中未定义

### 业务需求
用户可配置自有 WebDAV 服务器，将学习日记中的图片上传到 WebDAV 网盘存储。

### 完整实现方案

**前端改造（ProfilePage 中 WebDavSection）：**
1. 密码输入框使用 `type="password"` + 显示/隐藏切换
2. 保存时对密码进行客户端加密（使用 Web Crypto API 的 AES-GCM）
3. 加密密钥从用户密码派生（PBKDF2）
4. 「测试连接」按钮验证 WebDAV 可达性

**后端实现（新增 Edge Function `webdav-upload`）：**
```
POST /functions/v1/webdav-upload
Body: { file: Blob, path: string }
Response: { url: string }
```
- 使用 service_role 读取用户 WebDAV 配置
- 解密密码
- 通过 HTTP PUT 上传到用户 WebDAV 服务器
- 返回可访问 URL

**数据库变更：**
```sql
ALTER TABLE user_settings ADD COLUMN IF NOT EXISTS webdav_url TEXT;
ALTER TABLE user_settings ADD COLUMN IF NOT EXISTS webdav_user TEXT;
ALTER TABLE user_settings ADD COLUMN IF NOT EXISTS webdav_pass_enc TEXT;
```

## 3.3 每日推荐持久化（refreshDailyPick 写入被 RLS 拒绝）

### 现状
`refreshDailyPick` 尝试 upsert `daily_picks` 表，但该表写入策略为 service_role only。前端使用 anon key，写入被 RLS 拒绝（静默失败）。

### 业务需求
用户点击「换一批」时，新的每日推荐应持久化到数据库，确保所有用户看到相同的推荐内容。

### 完整实现方案

**方案A（推荐）：Edge Function**
```
POST /functions/v1/refresh-daily-pick
Body: { type: 'word' | 'sentence' | 'both' }
Auth: Bearer token (authenticated user)
```
- Edge Function 使用 service_role 写入 daily_picks
- 随机选择 enriched 词条 + 随机句子
- Upsert 到 daily_picks（onConflict: pick_date）

**方案B：RLS 策略调整**
```sql
CREATE POLICY "authenticated_can_write_daily_picks"
  ON daily_picks FOR ALL TO authenticated
  USING (true) WITH CHECK (true);
```
注意：方案B 允许多用户并发写入可能产生冲突，需配合 `ON CONFLICT DO UPDATE`。

## 3.5 学习统计可视化（StatsSection 功能不完整）

但缺少以下统计维度：
- 累计学习天数 / 最长连续天数
- 总查词量 / 总收藏量
- 各分类学习时长占比

### 业务需求

### 前端改造
在 `StatsSection.jsx` 中新增：
1. **概览卡片行**：累计天数 | 最长连续 | 总查词 | 总收藏
2. **月度热力图**：类似 GitHub contribution graph（12个月）
3. **分类占比饼图**：单词/语法/阅读/听力各占多少时间

### 后端改造
新增 RPC 函数高效聚合统计数据：
```sql
CREATE OR REPLACE FUNCTION get_user_stats_summary(p_user_id UUID)
RETURNS JSON AS $$
DECLARE result JSON;
BEGIN
  SELECT json_build_object(
    'total_days', COUNT(DISTINCT completed_date),
    'longest_streak', -- 需要计算最长连续日期序列
    'total_words', (SELECT COUNT(*) FROM user_recent_words WHERE user_id = p_user_id),
    'total_bookmarks', (SELECT COUNT(*) FROM user_bookmarks WHERE user_id = p_user_id),
    'total_study_minutes', COALESCE(SUM(study_minutes), 0)
  ) INTO result
  FROM user_checkin_completions
  WHERE user_id = p_user_id;
  RETURN result;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## 3.7 提醒邮件定时触发（send-reminder 存在但无调度）

### 现状
`send-reminder` Edge Function 已实现发送提醒邮件的逻辑，`ReminderPage.jsx` 有完整的提醒设置 UI（时间、频率、任务列表），但：
- 无定时触发机制（需要 cron job 或外部调度器）
- 前端「发送测试提醒」按钮可正常工作
- 用户设置的提醒时间/频率存储在 `user_settings` 中但无后端读取

### 业务需求
每天在用户设定的时间自动发送学习提醒邮件。

### 完整实现方案

**方案A：Supabase pg_cron（推荐）**
```sql
-- 启用 pg_cron 扩展（Supabase 内置）
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- 每天 UTC 01:00（CST 09:00）触发
SELECT cron.schedule(
  'send-daily-reminders',
  '0 1 * * *',
  $$ SELECT net.http_invoke(
    'https://zvemahqskgluhirzbcqu.supabase.co/functions/v1/send-reminder',
    'POST',
    headers := '{"Authorization": "Bearer ' || current_setting('app.settings.service_role_key') || '"}',
    body := '{"scheduled": true}'
  ) $$
);
```

**方案B：外部 cron 服务**
使用 GitHub Actions / Vercel Cron / cron-job.org 定时调用 Edge Function。

**Edge Function 改造：**
`send-reminder` 需要查询所有开启了提醒的用户，逐个发送邮件：
```ts
const { data: users } = await supabase
  .from('user_settings')
  .select('*, auth.users(email)')
  .eq('reminder_enabled', true)
```

## 3.8 单词书功能（WordBookPage 第4个tab）

### 现状
`WordBookPage.jsx` 有 4 个 tab：最近查词、单词夹、句子夹、单词书。前3个功能已实现，「单词书」tab 的 UI 存在但数据源为空。

### 业务需求
「单词书」是系统预设的分级词汇表（如「HSK泰语对照500词」「旅行必备200句」），用户可按书为单位学习。

### 实现方案

```sql
CREATE TABLE IF NOT EXISTS word_books (
  id BIGSERIAL PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT DEFAULT '',
  level TEXT DEFAULT 'beginner',
  cover_color TEXT DEFAULT '#5B8C7E',
  sort_order INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT now()

CREATE TABLE IF NOT EXISTS user_word_book_progress (
```

## 3.9 用户成就/徽章系统（无入口，完全未实现）

### 业务需求
通过学习里程碑激励用户：
- 连续打卡 7 天 → 「一周坚持」徽章
- 收藏 100 个词 → 「百词斩」徽章
- 完成一本单词书 → 「学以致用」徽章

### 实现方案

**数据库：**
```sql
CREATE TABLE IF NOT EXISTS user_achievements (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  badge_key TEXT NOT NULL,
  unlocked_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(user_id, badge_key)
);
```

**触发机制：** 在关键操作完成后检查是否满足解锁条件（打卡完成、收藏达到阈值等），满足则插入 achievement 记录并显示 toast 通知。
