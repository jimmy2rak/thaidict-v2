# 2026-07-20 工作日志

## 修复 /api/send-otp 限频逻辑反转（Brevo 邮件发不出）

### 现象
- 用户反馈：F12 控制台 `POST /api/send-otp 429 (Too Many Requests)`；认为邮件还是「Supabase 自带」。
- 同时控制台有一个 `content_script...startsWith` 报错 —— 那是浏览器扩展（content script），与本项目无关，可忽略。

### 排查结论
- OTP 验证码邮件路径 `/api/send-otp` 确实调用 `sendBrevoEmail`（Brevo），magic link `/api/send-magic-link` 也是 Brevo，注册 `/api/sign-up` 用 `email_confirm:true` 不发邮件。
- 全量 grep `src/` + `app/`：**无任何** `supabase.auth.signInWithOtp` / `signUp` 调用，应用从不走 Supabase 自带邮件器。
- 真正 bug：`check_otp_rate_limit(p_email)` 函数语义是 `not exists(60秒内记录)` → 返回 `true` 表示「可以发」。
  但路由写成 `if (limited) return 429`，把「可以发」误判为限频，**首条请求就被 429 拦截，Brevo 永远发不出**。用户因此收不到 Brevo 邮件，误以为是 Supabase 自带。

### 修复
- `app/api/send-otp/route.js`：改为 `allowed` 语义 + `if (!allowed) return 429`，并用 try/catch 包裹 RPC（函数缺失时放行，避免硬 500）。
- 不改动 SQL 函数（`not exists` = allowed 与新路由语义一致），无需跑迁移，Vercel 部署即生效。

### 提醒用户
- 确认 Vercel 环境变量 `BREVO_API_KEY` / `BREVO_SENDER_EMAIL` / `BREVO_SENDER_NAME` 已配置（缺失时 `sendBrevoEmail` 抛错 → 502，非 429）。当前 429 与此无关。

### 构建与推送
- `npm run build` 通过 → 提交并 `git push origin main` → Vercel 自动部署。

## /api/send-otp 502 具体错误：Brevo 返回 HTML 而非 JSON

### 现象（用户贴出的页面文字）
- `发送失败： Unexpected token '<', "<!DOCTYPE "... is not valid JSON`
- 这是 `brevo.js` 原代码在 `res.ok` 时直接 `res.json()` 解析到 HTML 页面抛出的错。

### 根因推测
- Brevo 接口返回了 HTML 页（而非 JSON），常见原因：
  1. 请求被重定向到登录/网关页（如 Vercel 出站被拦截、或 API Key 触发代理）；
  2. 端点返回 404/错误 HTML（路径或区域不对）；
  3. `BREVO_API_KEY` 值带多余引号/空格导致鉴权失败被跳转到 HTML 登录页。

### 已做
- `src/lib/brevo.js` 改为：先 `res.text()` 取原始响应，判断是否为 HTML；
  - 非 2xx：抛出 `Brevo HTTP {status}: {原始片段}`；
  - 2xx 但却是 HTML：抛出 `Brevo 返回了非 JSON（疑似网关/代理页）: {片段}`；
  - 否则 `JSON.parse(raw)`。
- 这样部署后页面会显示**具体 HTTP 状态 + HTML 前 200 字符**，便于定位。

### 用户下一步
- 重新触发一次发送，把新的「发送失败：」文字发我（应包含 HTTP 状态与 HTML 片段）。
- 同时核对 Vercel 里 `BREVO_API_KEY` 值没有多余引号/换行，且 Brevo 后台 Senders 中发件人已验证。

## /api/send-otp 仍空 502 → 改为并行 + maxDuration，彻底消除空 502

### 根因确认
- 空 502 = Vercel Function 在平台层被超时掐断（路由内 try/catch 来不及返回 JSON）。
- 旧路由：限频 RPC(5s) + 写库(5s) + Brevo(10s) **串行**，最坏 ~20s 远超 Hobby 10s 上限 → 网关/代理丢空 502。
- 用户已重传 BREVO_API_KEY / 发件人，故不是缺环境变量，而是网络调用叠加超时。

### 修复（重写 route）
- `export const maxDuration = 10`（显式声明 Hobby 上限）。
- 限频检查 `check_otp_rate_limit` 改为**后台触发、不 await**，不阻塞主流程。
- 写库 + Brevo 发信改为 `Promise.all` **并行**，总耗时 = max(写库, 发信) ≈ 8s，< 10s。
- insert 失败 Supabase 返回 `{error}` 不 reject，故单独 `insRes.error` 检查；Brevo 用 `.then(()=>null).catch(e=>e)` 捕获错误对象，避免 Promise.all 误判成功。
- 起始 `console.log('[send-otp] env:', {hasBrevoKey, hasSender, hasSupabaseUrl, hasServiceKey})` 确认变量注入情况（不打印值）。
- 超时/异常均返回带文字的 JSON 502，不再有空 502。

### 部署后预期
- 若 Brevo 从 Vercel 区域可达：正常收到邮件。
- 若 Brevo 不可达：页面显示 `发送失败：Brevo 发送超时（8000ms 未完成）`，可据此确认是 Brevo 网络问题 → 需换 SMTP/代理。
- 用户应在 Vercel Dashboard → Logs 查 `[send-otp] env` 确认四个变量均为 true。
