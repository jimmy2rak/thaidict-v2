# 2026-07-20 工作日志

## 修复 /api/send-otp 限频逻辑反转（Brevo 邮件发不出）

### 现象
- 用户反馈：F12 控制台 `POST /api/send-otp 429 (Too Many Requests)`；认为邮件还是「Supabase 自带」。
- 同时控制台有一个 `content_script...startsWith` 报错 —— 那是浏览器扩展（content script），与本项目无关，可忽略。

### 排查结论
- OTP 验证码邮件路径 `/api/send-otp` 确实调用 `sendBrevoEmail`（Brevo），magic link `/api/send-magic-link` 也是 Brevo，注册 `/api/sign-up` 用 `email_confirm:true` 不发邮件。
- 全量 grep `src/` + `app/`：**无任何** `supabase.auth.signInWithOtp` / `signUp` 调用，应用从不走 Supabase 自带邮件器。
- 真正 bug：`check_otp_rate_limit(p_email)` 函数语义是 `not exists(60秒内记录)` → 返回 `true` 表示「可以发」。
  但路由写成 `if (limited) return 429`，把「可以发」误判为限频，**首条请求就被 429 拦截，Brevo 永远发不出**。用户因此收不到 Brevo 邮件，误以为是 Supabase 自带。

### 修复
- `app/api/send-otp/route.js`：改为 `allowed` 语义 + `if (!allowed) return 429`，并用 try/catch 包裹 RPC（函数缺失时放行，避免硬 500）。
- 不改动 SQL 函数（`not exists` = allowed 与新路由语义一致），无需跑迁移，Vercel 部署即生效。

### 提醒用户
- 确认 Vercel 环境变量 `BREVO_API_KEY` / `BREVO_SENDER_EMAIL` / `BREVO_SENDER_NAME` 已配置（缺失时 `sendBrevoEmail` 抛错 → 502，非 429）。当前 429 与此无关。

### 构建与推送
- `npm run build` 通过 → 提交并 `git push origin main` → Vercel 自动部署。

## 修复 /api/send-otp 500：otp_codes 缺少 purpose 列

### 现象
- 用户点击「发送验证码」提示：
  `发送失败：Could not find the 'purpose' column of 'otp_codes' in the schema cache`
- F12：POST /api/send-otp 500 (Internal Server Error)

### 排查结论
- `/api/send-otp` 和 `/api/verify-otp` 都向 `otp_codes` 写入/查询 `purpose` 字段。
- 当前生产库 `otp_codes` 表没有 `purpose` 列；01-create-schema.sql 已有该列，但旧库未同步。

### 修复
- 新增 `supabase/migrations/06-add-purpose-to-otp-codes.sql`：
  - `alter table otp_codes add column if not exists purpose text not null default 'login';`
  - `create index if not exists idx_otp_email_purpose on otp_codes (email, purpose);`
  - 把旧记录 `purpose` 统一为 `'login'`。

### 用户下一步
- 在 Supabase SQL Editor 中执行 `supabase/migrations/06-add-purpose-to-otp-codes.sql` 后，/api/send-otp 即可正常写入并通过 Brevo 发信。

## 修复 /api/send-otp 500：otp_codes 缺少 type 列（not-null 约束）

### 现象
- 执行 06 后仍报错：`"otp_codes" violates not-null constraint` / `null value in column "type" of relation "otp_codes"`。
- 说明现有库的 otp_codes 表除了缺 purpose，还有一个 NOT NULL 的 `type` 列，而路由只写入了 email/code/purpose/expires_at。

### 修复
- 新增 `supabase/migrations/07-otp-codes-type-column.sql`：
  - `add column if not exists purpose text not null default 'login';`（幂等，兼容 06 未跑）
  - `add column if not exists type text not null default 'email';`
  - 回填旧记录，建 `(email, purpose)` 索引。
- `app/api/send-otp/route.js` 的 insert 增加 `type: 'email'`。

### 用户下一步
- 在 Supabase SQL Editor 执行 `supabase/migrations/07-otp-codes-type-column.sql`（06 已跑过也无妨，脚本幂等）。
- 若再报其他列的 not-null，按同样模式补列 + 在 insert 中给默认值即可。

## 修复 /api/send-otp 500：otp_codes.type 的遗留 CHECK 约束

### 现象
- 执行 07 后仍报错：`new row for relation "otp_codes" violates check constraint "otp_codes_type_check"`。
- 现有库 `otp_codes.type` 列带有一个 CHECK 约束，其允许值集合与路由写入的 `'email'` 不符（约束是旧 schema 遗留）。

### 修复
- 新增 `supabase/migrations/08-drop-otp-type-check.sql`：
  - `DO $$ ... alter table otp_codes drop constraint if exists otp_codes_type_check;`（幂等）。
- 路由已写入 `type: 'email'`，移除约束后即可正常插入。

### 用户下一步
- 在 Supabase SQL Editor 执行 `supabase/migrations/08-drop-otp-type-check.sql`。
