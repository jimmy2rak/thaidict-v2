# 2026-07-16 中泰词典工作日志

## 7. PyThaiNLP newmm 分词方案落地（commit 40f4680 已推）

### 需求
用户反馈例句分词仍大量出错/未分，截图中 `ของนาฬิกา`、`ควบคุมราคาสินค้า`、`มีประสิทธิภาพสูง` 等被过度合并。要求用 PyThaiNLP 的 newmm，并「把 newmm 直接导入数据库中」。

### 方案架构（三层）
1. **数据库金标准**：所有现有例句（dictionary / community_words / sentences）用 PyThaiNLP newmm 预分词，结果写入 `segmented` 字段；WordDetailPage 优先使用 `ex.segmented` 渲染，无需运行时计算。
2. **服务端实时兜底**：独立 Python 微服务 `services/thai-segment/` 运行 PyThaiNLP newmm；Next.js `/api/thai-segment` 优先调用它，服务不可用时降级为服务端 JS 分词器。
3. **新数据入库兜底**：`saveCommunityWord` / `addDictionaryWord` 在入库前调用 `fillSensesSegments`，对 AI 漏给 segmented 的例句自动补充分词。

### 全面自动拆词：词库兜底（commit 7330736 已推，取代手工逐条加词典方案）
- 用户明确：要【所有】短语库成语都拆，不要只靠 `custom_dict.txt` 手工 7 条。
- 根因定位：PyThaiNLP newmm 会把泰语成语/谚语当成一个词（单 token），无论加多少 custom_dict 都只能覆盖已知几条。
- 方案：`seg_utils` 新增 `load_wordlist_from_supabase`（拉 `dictionary.word` 6 万词集）+ `greedy_segment`（左到右最长匹配）+ `_maybe_break_idiom`（newmm 产出的长单 token 若 >=6 字符，用词库贪婪拆成已知成分词，需切成 >=2 个 >=2 字符的有效词才采用，避免误拆）。`segment()` 改为 `newmm → 长词兜底 → custom_dict 递归展开`。本地已验证：เข้าตามตรอกออกตามประตู→6词、ผีซ้ำด้ำพลอย→4词、เดินตามหลังผู้ใหญ่หมาไม่กัด→7词；普通已正确分词（如 ประเทศไทย）不受影响。
- `main.py`：新增 `_get_wordset()` 懒加载（优先 `/app/wordlist.txt` 持久化文件，否则从 Supabase 拉取并写回文件缓存）；`/segment`、`/legacy`、`/batch` 均传 `word_set`；`/health` 仅报告 `wordlist_size`（不触发加载，避免首次健康检查卡顿）。
- `segment_existing_data.py`：`main()` 启动时 `load_wordlist_from_supabase` 拉词集赋给全局 `WORDSET`；`segment_text` 改为复用 `seg_utils.segment`（带词库兜底）；新增 `ONLY_CATEGORY`（只处理该 category 的 sentences，如 idioms/buddhist/daily）。

### 【重大坑】Supabase REST 1000 行上限致词库只拉 987（commit ac98699 已推）
- 用户跑 generate_wordlist 只得 `dictionary 词数：987`（应约 6 万）。
- **根因**：Supabase/PostgREST 默认 `db-max-rows=1000`，单次请求最多返回 1000 行，`limit=2000` 无法突破。旧分页终止条件是「返回行数 < 2000 即末页」→ 第一页被截成 1000 行、去重后 987，直接误判结束。
- **同款 bug 也在 `seg_utils.load_wordlist_from_supabase`（batch=2000）**——所以即便容器带凭据，运行时词库也只会拉到约 1000 词，这就是为什么 7330736「词库兜底」方案一直没真正生效。
- **修复**：
  1. `generate_wordlist.py`：改用 PostgREST `Range` 头分页（`Range-Unit: items` + `Range: offset-(offset+999)`），PAGE=1000，以「返回 0/不足一页」判定结束；每页打印累计词数。
  2. `seg_utils.load_wordlist_from_supabase`：batch 默认 2000→1000（<=db-max-rows），终止条件不变但现在正确（每满页恰 1000）。
- **PITFALL 定律**：任何走 Supabase REST 全表分页的地方，batch 必须 <=1000，且终止条件用「返回空/不足一页」，绝不能用 batch>1000 + 「len<batch 即末页」。
- 用户服务器有个未推的本地 commit `adeefc5`（987 词 wordlist），git pull 远端只改 py 不动 wordlist.txt 不冲突；`generate_wordlist.py --force` 会覆盖旧文件重新生成。服务器 https push 需 PAT，但本机 docker build 读本地文件无需 push。
